# /manager-resume

You are taking over manager duties from a previous manager session. Do the following in order:

1. Parse `$ARGUMENTS` for the `handoff_id`. If missing or empty, tell the user "ERROR: /manager-resume requires a handoff_id" and stop.
2. Read `~/.claude/dockwright/handoffs/<handoff_id>.json`. If the file is missing, tell the user "ERROR: no handoff with id <handoff_id>" and stop. If `consumed_at` is non-null, tell the user "ERROR: handoff <handoff_id> already consumed at <consumed_at>" and stop.
3. Resolve `<your sid>` from the session id in system context / `$CLAUDE_CODE_SESSION_ID` (managers run under Claude Code). If you need to inspect it from shell, run `echo $CLAUDE_CODE_SESSION_ID`. Do not invent or synthesize a sid.
4. Call the `dockwright` MCP tool `become_manager_with_takeover` with `claude_sid=<your sid>`, `takeover_from=<the handoff's from_sid>`, and `handoff_id=<id>`. If it returns an error, surface it and stop. It returns your `name` and `domain` (and `runtime`, always `claude`).
4b. Paint your tmux tab so it shows your funny name + domain (the SessionStart hook can't do this for user-launched managers — they have no `CLAUDE_AGENT` env — and `become_manager_with_takeover` doesn't repaint). RUN, substituting `<name>`/`<domain>` from the return value:

```bash
SOCK="${DOCKWRIGHT_TMUX_SOCKET:-${CLAUDE_ORCH_TMUX_SOCKET:-dockwright}}"
[ -n "$TMUX_PANE" ] && tmux -L "$SOCK" rename-window -t "$TMUX_PANE" "<name> · <domain>" \
  && tmux -L "$SOCK" set-window-option -t "$TMUX_PANE" window-status-current-style "bg=#aa0066,fg=#ffffff" \
  && tmux -L "$SOCK" set-window-option -t "$TMUX_PANE" window-status-style "bg=#440022,fg=#ffffff"
```

(Mirrors `TmuxDriver.set_tab_title`/`set_tab_color`; colors match `MANAGER_TAB_COLOR` in `hooks.py`. Skip silently if `$TMUX_PANE` is unset (not running under tmux).)
4c. **REQUIRED — confirm the predecessor's pane is closed.** `become_manager_with_takeover` closes the predecessor's tmux pane automatically via `get_driver().close()` (tmux `kill-pane`). You MUST verify it actually closed:

```bash
SOCK="${DOCKWRIGHT_TMUX_SOCKET:-${CLAUDE_ORCH_TMUX_SOCKET:-dockwright}}"
tmux -L "$SOCK" list-panes -a -F '#{pane_id} #{window_name}'   # confirm no pane's window-name carries <predecessor-name> / the from_sid
# stray predecessor pane still open? close it: tmux -L "$SOCK" kill-pane -t <pane_id>
```

Check no pane remains for the predecessor (its window-name carried `<predecessor-name>`, or match the `from_sid`). If one is still open, kill it NOW with `tmux -L "$SOCK" kill-pane -t <pane_id>`. Do not skip this — two live managers for one handoff is a split-brain state.
5. Call the `dockwright` MCP tool `attach_existing` to enumerate workers and orphan questions (they carry forward from the old manager).
6. Load the manager agent definition from `~/.claude/agents/manager.md`. **This file is larger than a single `Read` can return (~80 KB / 600+ lines, over the ~25k-token single-Read cap), so one `Read` silently truncates it — the later sections just never arrive, with no error.** You MUST page it to EOF before acting: run `wc -l ~/.claude/agents/manager.md` to get the total line count N, then `Read` it in windows of ≤200 lines (`offset=1`, then `offset=201`, then `offset=401`, … each with `limit=200`) until a window reaches line N. **Do not act on a partial read** — if the last line you have read is below N, page again. Only once the whole file is in context, follow its operating principles for the rest of this session.
7. Load recent manager memory + the manager notebook for your domain. The handoff record carries the predecessor's `domain` and `manager_name`; the `become_manager_with_takeover` return value gives them back too. Run `bash -c 'TODAY=$(date +%s); for f in ~/.claude/dockwright/manager-memory/<domain>/*.md; do [ -e "$f" ] || continue; mtime=$(date -r "$f" +%s 2>/dev/null) || continue; age=$(( (TODAY - mtime) / 86400 )); [ "$age" -le 7 ] && echo "$mtime $f"; done | sort -rn | head -5 | while read -r _ f; do echo "MEMORY $f"; done; NB=~/.claude/dockwright/notebook/<domain>.md; if [ -s "$NB" ]; then SZ=$(wc -c < "$NB"); echo "NOTEBOOK $NB ($SZ bytes)"; [ "$SZ" -le 4096 ] || echo "NOTEBOOK_WARN [notebook >4KB ($SZ bytes) — archive resolved entries to notebook/archive/]"; fi'` — substitute `<domain>`. The command prints **paths only** — `Read` each printed `MEMORY <path>` and the `NOTEBOOK <path>` with the Read tool (Read paginates, so nothing is dropped). It used to `cat` the files inline, but 5 memory distills + the notebook can total ~35 KB and overflow the Bash inline-output cap, leaving only a ~2 KB preview in context — silently dropping memory and notebook content (this is why a boot could falsely report the notebook empty). The memory part loads the newest 5 files by mtime among those from the last 7 days (boot-cost cap). Fold these into your working context as predecessor context (your immediate predecessor's just-prepared distill is in there too — it's the newest file, so the cap never drops it). Skip silently if empty. The notebook part prints a pointer to your domain's planned/conditional-work agenda (`~/.claude/dockwright/notebook/<domain>.md` — see `~/.claude/agents/manager.md` § Manager notebook): if it printed, count its open entries (`## [ ]` headers), evaluate which look ripe (date conditions by inspection; others via the entry's named `check:` command — one cheap call each), and add "N planned entries, M look ripe" to the takeover brief, flagging any entry past its `review-by` for triage. If the size warning printed, archive resolved entries per the agent file. If the notebook is absent or empty, skip silently — no mention in the brief.
8. Surface a 3–5 line takeover brief to the user:
   - "Took over from <from_sid[:8]> (trigger: <trigger_reason>). N workers in flight, M pending questions. Loaded N memory files (newest 5 within last 7 days)."
   - The first paragraph of the handoff's `narrative_summary`, verbatim.
9. Arm the four Monitor tasks exactly as follows:
   1. Arm a `Monitor` task with command `while true; do dockwright monitor questions; sleep 2; done` for worker questions. Identity resolves per-scan; no substitution. The monitor watches ONLY `questions/<resolved-name>/`, while legacy flat `questions/<id>.json` files remain visible through `list_pending_questions(manager_sid=None)` for recovery.
   2. Arm a second `Monitor` task with command `while true; do dockwright monitor turn-ends; sleep 5; done` — the **silent-finish detector**. It does NOT ping per turn-end: routine turn-ends are suppressed (done events, questions, and stalls each have their own lane), and the scan emits only `FINISHED_SILENTLY <name>: <summary tail>` when a worker's turn ended ≥2min ago (grace, env `CLAUDE_ORCH_TURN_END_GRACE_SEC`) with no follow-up turn, no done event, and no pending question — i.e. it probably finished without calling `worker_done` (`(session exited)` variant when the session is gone). First line per lull immediate; repeats only at ladder rungs. Identity (which manager YOU are) is resolved per-scan via `TMUX_PANE` → PPID-walk → loud-fail; no substitution into the Monitor command. On a `FINISHED_SILENTLY` line, verify via `get_worker_summary` / `list_workers` and harvest, re-instruct, or close the worker. The monitor watches ONLY your own bucket (`turn-ends/<resolved-name>/`) — null-parent worker events in `_unscoped/` are not surfaced (recovery via `_backfill_legacy_workers` on single-manager boot). Repeat pages for the same lull are ladder-rate-limited (15/30/60min…, cap 4h; resets on re-instruction / done / session-exit) — one lull pages once immediately, then only on the ladder.
   3. Arm a third `Monitor` task with command `while true; do dockwright monitor done; sleep 2; done` for explicit `worker_done` events. Identity resolves per-scan; no substitution. The monitor watches ONLY `done/<resolved-name>/` — same `_unscoped/` invisibility as turn-ends. On each event the scan emits `<worker_name> done: <summary>` straight from the done file payload.
   4. Arm a fourth `Monitor` task with command `while true; do dockwright monitor stale; sleep 60; done` for stuck workers and stale questions. Identity resolves per-scan; `monitor stale` internally runs the packaged stale monitor (`python -m dockwright.stale_monitor --manager <resolved-name>`), scoping the scan to your own workers — restoring parity with `manager.md` (previously this step called `stale_monitor.py` without `--manager`, making it global; the new CLI resolves identity internally and passes the flag). STALE_PROCESSING / STALE_QUESTION / AUTOCLOSED events from peer managers never fire here. AUTOCLOSED lines arrive batched — piggybacked onto the next wake from any lane, or flushed within 30min.
10. From this point on, the user is in manager mode. Confirm with: "Manager active (resumed from <from_sid[:8]>)."
