# /manager

You are entering **manager mode**. Optional `$ARGUMENTS` selects a domain (default `general`); domains scope memory pools.

Do the following in order:

1. Parse `$ARGUMENTS` for a single token — that's `<domain>`. If empty, use `general`.
2. Resolve `<your sid>` from the session id in system context / `$CLAUDE_CODE_SESSION_ID` (managers run under Claude Code). If you need to inspect it from shell, run `printenv CLAUDE_CODE_SESSION_ID`. Do not invent or synthesize a sid.
3. Call `dockwright` MCP tool `become_manager` with `claude_sid=<your sid>`, session_id if available, and `domain=<domain>`. It returns `{ok, name, domain, runtime}` (`runtime` is always `claude`). The `name` is your auto-rolled funny `<adjective>-<animal>` handle, unique across all active records. Save it for use in subsequent tool calls (most tools take an optional `manager_sid=<your sid>`). It also paints your tmux tab (name · domain) and runs preflight cleanup server-side — surface the returned `preflight` line to the user if non-empty (older orchestrator builds may omit the field; then skip silently).
3b. Model check (managers are pinned to Opus 1M on the recreate/recovery lanes, but a user-launched `/manager` inherits the session default): if this session's model is not an Opus model (your system prompt states the model id, e.g. `claude-opus-4-8`), tell the user: "This manager is running on <model>; the manager lane is specced for Opus 1M — consider `/model opus[1m]` or relaunching." Then continue — it's a warning, not a gate.
4. Call `dockwright` MCP tool `attach_existing(manager_sid=<your sid>)`. If it errors, the orchestrator MCP isn't installed — surface the error and stop.
5. Read the result. Tell the user: "Manager mode active as **<name>** (domain: <domain>). Re-attached to N workers. M questions pending."
6. Load the manager agent definition from `~/.claude/agents/manager.md`. **This file is larger than a single `Read` can return (~80 KB / 600+ lines, over the ~25k-token single-Read cap), so one `Read` silently truncates it — the later sections just never arrive, with no error.** You MUST page it to EOF before acting: the total line count N is the `AGENT_LINES <N>` line printed by `dockwright boot-brief` (step 6b below). `Read` the file in windows of ≤200 lines (`offset=1`, then `offset=201`, then `offset=401`, … each with `limit=200`) until a window reaches line N. **Do not act on a partial read** — if the last line you have read is below N, page again. Only once the whole file is in context, follow its operating principles for the rest of this session.
6b. Load recent manager memory + the manager notebook for your domain: run `dockwright boot-brief --domain <domain>` (substitute `<domain>` with the domain you registered as). It prints AGENT_LINES plus **paths only** — `Read` each printed `MEMORY <path>` and the `NOTEBOOK <path>` with the Read tool (Read paginates, so nothing is dropped). It prints pointers rather than file contents because 5 memory distills + the notebook can total ~35 KB and overflow the Bash inline-output cap, leaving only a ~2 KB preview in context — silently dropping memory and notebook content (this is why an old inline-`cat` boot could falsely report the notebook empty). The `MEMORY` pointers are the newest 5 files by mtime among those from the last 7 days (a busy week can produce 15+ files; the cap keeps boot cost bounded). If any printed, these are distilled summaries of recent manager sessions in this domain (your peers + predecessors) — fold them into your working context. Surface "Loaded N memory files (newest 5 within last 7 days)." in your startup brief. If no `MEMORY` line printed, skip silently. The `NOTEBOOK` pointer is your domain's planned/conditional-work agenda (`~/.claude/dockwright/notebook/<domain>.md` — see `~/.claude/agents/manager.md` § Manager notebook): if it printed, count its open entries (`## [ ]` headers), evaluate which look ripe (date conditions by inspection; others via the entry's named `check:` command — one cheap call each), and add "N planned entries, M look ripe" to the startup brief, flagging any entry past its `review-by` for triage. If boot-brief printed a `NOTEBOOK_WARN` line, archive resolved entries per the agent file. If no `NOTEBOOK` line printed, skip silently — no mention in the brief.
7. Arm the four Monitor tasks exactly as follows:
   1. Arm a `Monitor` task with command `while true; do dockwright monitor questions; sleep 2; done` for worker questions. Identity resolves per-scan; no substitution. The monitor watches ONLY `questions/<resolved-name>/`, while legacy flat `questions/<id>.json` files remain visible through `list_pending_questions(manager_sid=None)` for recovery. On each ping, call `list_pending_questions(manager_sid=<your sid>)` to relay your own workers' pending questions.
   2. Arm a second `Monitor` task with command `while true; do dockwright monitor turn-ends; sleep 5; done` — the **silent-finish detector**. It does NOT ping per turn-end: routine turn-ends are suppressed (done events, questions, and stalls each have their own lane), and the scan emits only `FINISHED_SILENTLY <name>: <summary tail>` when a worker's turn ended ≥2min ago (grace, env `CLAUDE_ORCH_TURN_END_GRACE_SEC`) with no follow-up turn, no done event, and no pending question — i.e. it probably finished without calling `worker_done` (`(session exited)` variant when the session is gone). First line per lull immediate; repeats only at ladder rungs. Identity (which manager YOU are) is resolved per-scan via `TMUX_PANE` → PPID-walk → loud-fail; no substitution into the Monitor command. On a `FINISHED_SILENTLY` line, verify via `get_worker_summary` / `list_workers` and harvest, re-instruct, or close the worker. The monitor watches ONLY your own bucket (`turn-ends/<resolved-name>/`) — null-parent worker events in `_unscoped/` are not surfaced (recovery via `_backfill_legacy_workers` on single-manager boot). Repeat pages for the same lull are ladder-rate-limited (15/30/60min…, cap 4h; resets on re-instruction / done / session-exit) — one lull pages once immediately, then only on the ladder.
   3. Arm a third `Monitor` task with command `while true; do dockwright monitor done; sleep 2; done` for explicit `worker_done` events. Identity resolves per-scan; no substitution. The monitor watches ONLY `done/<resolved-name>/` — same `_unscoped/` invisibility as turn-ends. On each event the scan emits `<worker_name> done: <summary>` straight from the done file payload.
   4. Arm a fourth `Monitor` task with command `while true; do dockwright monitor stale; sleep 60; done` for stuck workers and stale questions. Identity resolves per-scan; `monitor stale` internally runs the packaged stale monitor (`python -m dockwright.stale_monitor --manager <resolved-name>`), scoping the scan to your own workers. STALE_PROCESSING / STALE_QUESTION / AUTOCLOSED events from peer managers never fire here. AUTOCLOSED lines arrive batched — piggybacked onto the next wake from any lane, or flushed within 30min.
8. From this point on, the user is in manager mode. Confirm with: "Manager active. Tell me what to spawn."
